# ADR 022: Calico CNI Provider for Kubernetes Network Policies

## Status

Accepted (January 2026)

## Date

2026-01-29

## Context

Kubernetes Network Policies provide essential security controls for Zero-Trust architectures (see `docs/arc42/security-architecture.md`). However, **Network Policies require a CNI (Container Network Interface) provider that implements the NetworkPolicy API**.

### Problem

kind uses **kindnetd** as the default CNI, which **does NOT support Network Policies**:

```bash
$ kubectl apply -f network-policies.yaml
# → Policy objects created but NOT enforced

$ kubectl get networkpolicies
NAME                 POD-SELECTOR    AGE
default-deny-all     <none>          1m
# → Appears to work, but has NO EFFECT
```

**Consequence:** Network Policies silently ignored, no network segmentation, security risk.

### Requirements

1. **Network Policy Support**: Full NetworkPolicy API implementation
2. **kind Compatibility**: Must work with kind clusters
3. **Multi-Node Support**: Support 3-node cluster (1 control-plane + 2 workers)
4. **Layer 3 Networking**: IP-based policies (not just Layer 2)
5. **Easy Management**: Simple installation and troubleshooting
6. **Production-Ready**: Used in real production environments
7. **Zero-Trust Ready**: Support default-deny and least-privilege patterns

### CNI Candidates

1. **Calico**: Full-featured CNI with BGP, IPAM, Network Policies
2. **Cilium**: eBPF-based CNI with advanced features
3. **Weave Net**: Simple overlay network with Network Policy support
4. **Flannel + Calico**: Flannel for routing, Calico for policies

## Decision

**We deploy Calico v3.27.0 as the CNI provider to enable Kubernetes Network Policies.**

## Implementation

### Calico Installation

**Challenge:** Calico manifest references external Docker registries (quay.io, docker.io), requires workaround for kind.

**Solution:** Pre-load images + kubectl apply

```bash
# Script: k8s/kind/fix-calico-remote.sh

# 1. Pre-load images to kind nodes
CALICO_IMAGES=(
  "quay.io/calico/node:v3.27.0"
  "quay.io/calico/cni:v3.27.0"
  "quay.io/calico/kube-controllers:v3.27.0"
  "docker.io/calico/node:v3.27.0"
  "docker.io/calico/cni:v3.27.0"
  "docker.io/calico/kube-controllers:v3.27.0"
)

for image in "${CALICO_IMAGES[@]}"; do
  docker pull "$image"
  kind load docker-image "$image" --name erechnung
done

# 2. Apply Calico manifest
kubectl apply -f https://raw.githubusercontent.com/projectcalico/calico/v3.27.0/manifests/calico.yaml

# 3. Wait for Calico to be ready
kubectl wait --for=condition=ready pod -l k8s-app=calico-node -n kube-system --timeout=300s
```

**Components:**

- **calico-node:** DaemonSet (1 pod per node = 3 pods)
- **calico-kube-controllers:** Deployment (1 replica)

### Network Policy Implementation

**File:** `k8s/kind/network-policies.yaml`

**Zero-Trust Base:**

```yaml
# Deny all traffic by default
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: erechnung
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
```

**Allow DNS (required for all pods):**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns-access
  namespace: erechnung
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
    ports:
    - protocol: UDP
      port: 53
```

**Application-Specific Policies (12 total):**

- Ingress → API Gateway
- API Gateway → Django-Web
- Django + Celery → PostgreSQL
- Django + Celery → Redis
- Egress for external HTTPS (ZUGFeRD XML download)

## Rationale

### Why Calico?

**1. Production-Grade Network Policies:**

- Used by thousands of production clusters
- CNCF graduated project
- Trusted by enterprises globally

**2. Comprehensive Feature Set:**

- Full NetworkPolicy API support
- Layer 3/4 policy enforcement
- Layer 7 (Application Layer) policies (advanced)
- Global network policies (Calico-specific extension)

**3. kind Compatibility:**

- Well-documented for kind usage
- Works with kind's Docker network
- No special patches required (with image pre-load workaround)

**4. Performance:**

- iptables-based enforcement (fast)
- No performance overhead for allowed traffic
- Scales to large clusters (tested with 1000+ nodes)

**5. Observability:**

- Detailed policy logs
- Flow logs for debugging
- Integration with monitoring tools

### Why Not Cilium?

**Cilium Advantages:**

- eBPF-based (potentially faster)
- Advanced L7 policies
- API-aware security

**Cilium Disadvantages:**

- ❌ More complex setup
- ❌ Requires kernel 4.9+ with eBPF support
- ❌ Heavier resource usage
- ❌ Overkill for our needs (we don't need L7 policies yet)

**Decision:** Calico sufficient for current requirements, simpler setup.

### Why Not Weave Net?

**Weave Limitations:**

- Slower performance (overlay network overhead)
- Less feature-rich than Calico
- Smaller community
- Project less actively developed

### Why Not Flannel + Calico?

**Hybrid Approach:**

- Flannel for CNI (routing)
- Calico for Network Policies (security)

**Why Not:**

- More complex (two systems to manage)
- Calico alone provides both routing and policies
- Unnecessary complexity for our use case

## Network Policy Strategy

### Zero-Trust Architecture

**1. Default-Deny:**

```yaml
# Block all traffic by default
spec:
  podSelector: {}  # Applies to all pods
  policyTypes:
  - Ingress
  - Egress
```

**2. Least-Privilege Access:**

```yaml
# Allow only specific traffic
spec:
  podSelector:
    matchLabels:
      app: django-web
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: api-gateway
    ports:
    - protocol: TCP
      port: 8000
```

**3. DNS Always Allowed:**

```yaml
# All pods need DNS resolution
egress:
- to:
  - namespaceSelector:
      matchLabels:
        kubernetes.io/metadata.name: kube-system
  ports:
  - protocol: UDP
    port: 53
```

### Policy Layers

**Layer 1: Default-Deny** (baseline security)
**Layer 2: Infrastructure** (DNS, monitoring)
**Layer 3: Ingress** (external → cluster)
**Layer 4: Application** (inter-pod communication)
**Layer 5: Database** (backend → database)
**Layer 6: Egress** (cluster → external)

## Consequences

### Positive

- ✅ **Network Policies Work:** Policies enforced, not silently ignored
- ✅ **Zero-Trust Architecture:** Default-deny with least-privilege access
- ✅ **Production-Ready:** Same CNI as real production clusters
- ✅ **Observability:** Can debug policy issues with Calico logs
- ✅ **Scalable:** Supports multi-node clusters, horizontal scaling
- ✅ **Feature-Rich:** Advanced policies available if needed later

### Negative

- ⚠️ **Image Pre-Loading:** Requires `kind load` workaround (one-time)
- ⚠️ **Complexity:** More complex than kindnetd (default CNI)
- ⚠️ **Resource Usage:** Calico pods consume ~200MB RAM per node
- ⚠️ **Debugging:** Policy troubleshooting can be challenging initially

### Neutral

- **iptables-Based:** Uses Linux iptables (standard, well-understood)
- **BGP Routing:** Calico supports BGP (not used in kind, but available for production)

## Troubleshooting

### Common Issues

**1. Pods Stuck in Init:**

```bash
# Calico CNI binary missing in kind nodes
# Fix: Pre-load images with `kind load docker-image`
```

**2. Network Policies Not Enforced:**

```bash
# kindnetd still active (wrong CNI)
# Fix: Ensure Calico DaemonSet running on all nodes
kubectl get ds -n kube-system calico-node
```

**3. No Internet Access from Pods:**

```bash
# Default-deny blocks egress
# Fix: Add egress policy for external HTTPS
kubectl apply -f network-policies.yaml
```
