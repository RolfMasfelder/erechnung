# Installation Guide

This guide covers how to install and deploy the eRechnung application. Two deployment modes are supported:

| Mode | Target | Scaling | Setup Effort |
|------|--------|---------|--------------|
| **Docker Compose** | Single host, small teams (1–10 users) | Vertical | Low |
| **Kubernetes (k3s)** | On-premise or cloud, larger teams | Horizontal | Medium |

Both modes run the same application code and produce identical results.

---

## Prerequisites

| Requirement | Minimum Version | Notes |
|-------------|-----------------|-------|
| Docker | 24+ | With Docker Compose v2 plugin (`docker compose`) |
| Git | 2.30+ | For cloning the repository |
| Linux host | Debian 12 / Ubuntu 22.04+ | macOS and WSL2 also work for Docker mode |

> **Note:** No local Python, Node.js, or npm installation is required. Everything runs inside containers.

## 1. Docker Compose Deployment

### 1.1 Clone and Configure

```bash
git clone <repository-url>
cd eRechnung_Django_App

# Generate secure secrets (recommended)
cd scripts && ./generate-secrets.sh && cd ..

# Or manually: copy template and edit
cp .env.example .env
# Edit .env — set at least DJANGO_SECRET_KEY and DB_PASSWORD
```

The `.env` file controls all configuration. Key variables:

| Variable | Purpose | Example |
|----------|---------|---------|
| `DJANGO_SECRET_KEY` | Django cryptographic signing | Auto-generated by `generate-secrets.sh` |
| `DB_PASSWORD` | PostgreSQL password | Auto-generated by `generate-secrets.sh` |
| `DJANGO_DEBUG` | Debug mode | `false` for production |
| `DJANGO_ALLOWED_HOSTS` | Allowed hostnames | `localhost,127.0.0.1` |
| `REDIS_PASSWORD` | Redis password (optional) | Leave empty for dev |

See `.env.example` for the full list of available variables.

### 1.2 Start the Application

**Option A — Quick Start Script (recommended for first run):**

```bash
cd scripts && ./start_app.sh
```

This script handles migrations, superuser creation, and static file collection automatically.

Use `--dev` for development mode with hot-reload volume mounts:

```bash
cd scripts && ./start_app.sh --dev
```

**Option B — Manual Start:**

```bash
# Backend (Django + PostgreSQL + Redis + API Gateway)
docker compose up -d --build

# Frontend development server (separate terminal)
docker compose -f docker-compose.frontend.yml up -d
```

### 1.3 Access Points

| URL | Service | Notes |
|-----|---------|-------|
| http://localhost:5173 | Frontend (Vue.js) | Development server with HMR |
| http://localhost:8000/admin/ | Django Admin | Direct backend access |
| http://localhost:8000/api/docs/ | Swagger UI | Interactive API documentation |
| https://localhost | API Gateway | HTTPS reverse proxy (when gateway is running) |

Default credentials after first run: `admin` / `adminpassword` (change immediately in production).

### 1.4 Port Reference

| Port | Service | Compose File | Exposed |
|------|---------|-------------|---------|
| 80 | API Gateway (nginx) | `docker-compose.yml` | Yes — HTTP → HTTPS redirect |
| 443 | API Gateway (nginx) | `docker-compose.yml` | Yes — HTTPS reverse proxy |
| 5173 | Frontend (Vite) | `docker-compose.frontend.yml` | Yes — Dev server |
| 8000 | Django Backend | `docker-compose.yml` | Yes — Direct API access |
| 5432 | PostgreSQL | `docker-compose.yml` | No — Internal only |
| 6379 | Redis | `docker-compose.yml` | No — Internal only |

### 1.5 Docker Compose File Overview

| File | Purpose |
|------|---------|
| `docker-compose.yml` | Base configuration (backend + DB + Redis + gateway) |
| `docker-compose.frontend.yml` | Frontend dev server |

| `docker-compose.production.yml` | Production configuration |

## 2. Kubernetes (k3s) Deployment

The k3s deployment provides horizontal scaling, Ingress-based routing, and production-grade security features. It runs on a dedicated server (physical or VM) in your local network or cloud.

### 2.1 Server Prerequisites

| Requirement | Details |
|-------------|---------|
| Linux server | Debian 12 / Ubuntu 22.04+ with 4 GB+ RAM |
| Network | Static IP or DHCP reservation for the k3s node |
| SSH access | For remote management |
| Helm 3 | For installing add-ons (Linkerd, Falco, monitoring) |

### 2.2 Install k3s

```bash
# On the target server
cd scripts && ./setup-k3s-local.sh
```

This script installs k3s, creates the namespace, and deploys all manifests from `infra/k8s/k3s/`.

### 2.3 Configure Access

After installation, copy the kubeconfig to your workstation:

```bash
export KUBECONFIG=~/.kube/config-k3s
```

The application is accessible via the Ingress controller at the server's IP address. Configure this IP in:

- `infra/k8s/k3s/ingress.yaml` — set the host or IP for your environment
- Your DNS or `/etc/hosts` — map a hostname to the server IP if desired

> **Note:** The k3s manifests reference the server IP in Ingress rules. Update these to match your own network — the default values are specific to the development environment and will not work elsewhere.

### 2.4 Kubernetes Infrastructure

Manifests are located in `infra/k8s/`:

```
infra/k8s/
├── api-gateway-deployment.yaml
├── api-gateway-service.yaml
├── django-backend-deployment.yaml
├── django-backend-service.yaml
├── ingress.yaml
├── k8s-erechnung.yaml         # All-in-one manifest
├── k3s/                        # k3s-specific configs
└── kind/                       # Local testing with kind
```

### 2.5 Optional Security Add-ons

These can be installed after the base deployment:

```bash
# Linkerd service mesh (mTLS between all services)
cd scripts && ./setup-linkerd-k3s.sh

# Falco runtime security monitoring
cd scripts && ./setup-falco-k3s.sh

# Kubernetes API audit logging
cd scripts && ./setup-k3s-audit-logging.sh
```

See [SECURITY_IMPLEMENTATION.md](SECURITY_IMPLEMENTATION.md) for details on security phases and features.

## 3. Secrets Management

### Generate Secrets

```bash
# Docker Compose secrets
cd scripts && ./generate-secrets.sh

# Also generate Kubernetes secrets
cd scripts && ./generate-secrets.sh --k8s
```

### Rotate Secrets

```bash
# Docker Compose (creates backup before rotating)
cd scripts && ./generate-secrets.sh --rotate

# Kubernetes (rotates DB password + restarts pods)
export KUBECONFIG=~/.kube/config-k3s
cd scripts && ./rotate-k8s-secrets.sh

# Preview without changes
cd scripts && ./rotate-k8s-secrets.sh --dry-run
```

> **Important:** `.env` contains real secrets and is excluded from version control via `.gitignore`. Use `.env.example` as a reference template.

## 4. Monitoring Stack (Optional)

### Docker Compose

```bash
docker compose -f docker-compose.yml -f docker-compose.monitoring.yml up -d
```

| Port | Service | Access |
|------|---------|--------|
| 3000 | Grafana | http://localhost:3000 (bound to 127.0.0.1) |
| 9090 | Prometheus | http://localhost:9090 (bound to 127.0.0.1) |
| 3100 | Loki | Internal — log aggregation |

Monitoring ports are bound to `127.0.0.1` and only accessible from the local host. Exporter ports (9121, 9187) are not exposed — Prometheus accesses them via the internal Docker network.

### Kubernetes

Grafana and Prometheus are accessible via Ingress subpaths at the k3s server IP:

- `http://<k3s-server-ip>/grafana/`
- `http://<k3s-server-ip>/prometheus/`

Monitoring UIs are secured with NetworkPolicies — only the Ingress controller can reach the pods.

## 5. Troubleshooting

### Static File Permissions

```bash
mkdir -p static media
chmod -R 755 static media
```

### Database Connection Errors

- Verify the PostgreSQL container is running: `docker compose ps`
- Check that `DB_*` variables in `.env` match the `POSTGRES_*` variables
- Both sets must agree: `DB_NAME` = `POSTGRES_DB`, `DB_USER` = `POSTGRES_USER`, etc.

### Migration Errors

```bash
docker compose exec web python project_root/manage.py migrate
```

If migrations fail due to missing defaults, see [DEVELOPER_GUIDE.md](DEVELOPER_GUIDE.md) for Django migration handling strategies.

### Container Won't Start

```bash
docker compose logs web     # Check backend logs
docker compose logs db      # Check database logs
docker compose logs redis   # Check Redis logs
```
